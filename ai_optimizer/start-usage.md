# Как использовать start.md

Инструкция к [start.md](./start.md) — bootstrap AI rules, документации и Cursor dispatcher для любого репозитория.

---

## Новый проект

1. **Скопируй** `E:\GitHub\start.md` в корень репозитория (или держи открытым рядом).
2. **Открой Cursor** в этом проекте → **Agent mode**.
3. **Вставь** промпт:

```text
Прочитай start.md в корне и выполни PART A + B + C.
Не меняй код приложения — только rules, docs и .cursor/*.
В конце — отчёт по шаблону из start.md.
```

Для маленького репо можно сократить:

```text
Прочитай start.md. Режим MVP: PART A + AI_CONTEXT + DECISIONS + PART C.
Не меняй код приложения. Отчёт по шаблону из start.md.
```

4. **Проверь отчёт** — пути в `AI_CONTEXT.md` должны быть реальными (не выдуманными).
5. **Закоммить** созданные файлы (`.cursorrules`, `AI_CONTEXT.md`, `.cursor/`, …).
6. **Работай как обычно** — широкие запросы («UI + deploy») агент разложит через dispatcher.

| Режим | Когда | Что создавать |
|-------|-------|---------------|
| **MVP** | Малый проект, быстрый старт | PART A + `AI_CONTEXT` + `DECISIONS` + PART C |
| **Full** | Большой репо, нужна полная документация | PART A + весь PART B + PART C (см. start.md) |

## Существующий проект

Здесь важно **не перезаписать** то, что уже есть.

1. **Посмотри**, что уже есть:
   - `.cursorrules`, `AI_RULES.md`, `AGENTS.md`, `CLAUDE.md`
   - `README`, `ARCHITECTURE.md`, другие `*.md`
2. **Скопируй** `start.md` в корень **или** укажи агенту путь `E:\GitHub\start.md`.
3. **Промпт с merge:**

```text
Прочитай E:\GitHub\start.md (или start.md в корне).
Проанализируй существующие AI/docs файлы.
Выполни только недостающее: merge, не перезаписывай существующие инструкции.
Обнови AI_CONTEXT.md и добавь Request routing + dispatcher (PART C).
Не трогай код приложения.
Отчёт: что было / что создано / что пропущено и почему.
```

4. **Безопаснее в два шага:**
   - **Ask mode:** «Что уже есть, что добавить из start.md?»
   - **Agent mode:** только недостающие файлы и merge.

5. **Проверь вручную:**
   - `AI_CONTEXT.md` — актуальные entry points, не копия README
   - dispatcher rule не конфликтует с твоими правилами
   - `.cursorignore` дополняет, а не ломает свой workflow
6. **Коммит** отдельным коммитом («AI bootstrap»), не смешивать с feature-работой.

Если docs уже есть — агент **дополняет** секции и ссылки в `AI_CONTEXT`, а не пересоздаёт всё.

## Сравнение

| | Новый проект | Существующий |
|---|--------------|--------------|
| Цель | С нуля поднять AI-инфраструктуру | Dispatcher + контекст без поломки |
| Docs | Full или MVP — создать | Merge / только пробелы |
| `.cursorrules` | Создать | Дополнить, не затирать |
| Риск | Низкий — обычно пусто | Перезапись старых rules/docs |
| После | Коммит → работа | Коммит → точечные обновления |

---

## Нужно ли держать start.md в репо

- **Не обязательно** — это одноразовый bootstrap.
- После настройки можно **удалить** из репо или положить в `docs/` «на память».
- **Жить должны:** `AI_CONTEXT.md`, `DECISIONS.md`, `.cursor/rules/`, `.cursor/skills/`, `.cursorignore`.

---

## Когда обновлять вручную

Обновляй (или попроси агента обновить) при изменении:

- deploy-скриптов и prod URL
- auth / OAuth / session
- главных модулей и integrations
- sync-политик (что уходит во внешние API, что только локально)

Файлы: **`AI_CONTEXT.md`** (routing, critical paths), **`DECISIONS.md`** (решения «почему так»).

## Примеры запросов после bootstrap

```text
Настрой мобильный UI, заодно проверь API и деплой.
→ план → UI → API → deploy отдельно

[multi] Фильтр на главной + миграции
→ явный triage

Почини 401 в src/.../auth.ts
→ без лишнего triage
```
